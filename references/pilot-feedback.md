# 小样本试跑反馈机制

全量筛选前必须先跑 3-5 份 pilot，除非用户明确说 JD 已经完全确认且愿意直接全量跑。

## 为什么要 pilot

HR 的真实筛选标准经常不完全写在 JD 里。pilot 的目的不是省时间，而是把隐藏标准显性化：

- “这个人我其实不想约”的原因是什么？
- “这个人虽然 JD 没写，但我愿意见”的原因是什么？
- 哪些条件是硬性要求，哪些只是加分？
- AI 是太宽、太严，还是看错重点？

## 小样本试跑后问用户什么

把前 3-5 份结果给用户看后，问：

1. `推荐` 里有没有你觉得不该推荐的人？为什么？
2. `不推荐` 里有没有你觉得被误杀的人？为什么？
3. `备选` 的数量和质量是否合适？
4. 一票否决项有没有执行到位？
5. 地点、年限、语言、行业、薪资这些硬条件有没有看准？
6. 接下来要更严格、更宽松，还是保持当前口径？

## Excel 反馈列

`筛选总表` 会预留这些列，用户可以直接在 Excel 里填写。它们主要用于抽检和纠偏，不要求每一行都填写；如果人工判断和 AI 判断基本一致，通常可以留空，用户也可以写“认可”作为明确确认：

| 列名 | 怎么填 |
|---|---|
| 人工初筛结果 | 用户对模型判断的直接看法，尤其是不同意 `AI 初筛结果` 时填写，例如“不该推荐”“其实一般”“被误杀”“需要复核”；“认可”只是可选确认项 |
| 人工初筛判断依据 | 用自然语言说明原因 |

这两列位于 `AI 初筛结果` 后面。Excel 会用浅黄色标出可填写区域；`人工初筛结果` 会提供常用下拉选项，但用户也可以直接输入自己的话。

示例：

```text
人工初筛结果：其实一般，不该放推荐
人工初筛判断依据：候选人是纯研发，没有任何客户、销售、市场或内容经验，不应该进推荐。
```

## 读取 Excel 反馈并重评

用户在 Excel 里填完反馈后，运行：

```bash
python3 scripts/resume_screening_pipeline.py score-only \
  --resumes ./resumes \
  --jd ./job_requirements.md \
  --work ./work \
  --output ./results \
  --feedback-file ./results/resume_screening_results.xlsx \
  --workers 2
```

脚本会读取 `筛选总表` 里的 `人工初筛结果` 和 `人工初筛判断依据`，把它写入每个候选人的 `human_feedback`，再让模型重新评分。模型会自己对比上一轮 `AI 初筛结果` 和人工反馈，判断是否高估、低估、误杀或基本正确；但仍然必须基于简历证据判断，不能编造事实。

面向 HR 时，不要让用户理解命令。可以这样说：

```text
你只需要保存 Excel，然后告诉我“我已经填好人工初筛结果了，请按这个 Excel 重评”。
我会读取这个文件，把你的反馈转成筛选标准修正，再重新评分。
```

## 如何把反馈写回 JD

不要只在对话里记住反馈，也不要只做一次性重评。把可复用的反馈追加到 `job_requirements.md` 的“筛选优先级”“一票否决项”或“小样本试跑后修正”下，例如：

```markdown
## 小样本试跑后修正
- 不要把纯技术研发背景、没有任何 GTM/客户/内容经验的人标为推荐。
- 如果不在湾区，但能明确每周到 SF 参加活动，可以进入备选。
- 有英文内容作品链接的人优先级上调。
```

## 何时重跑

- 如果只改了评分口径，优先使用 `score-only` 重新评分，不需要重新抽取简历。
- 如果用户在 Excel 里填了反馈，使用 `score-only --feedback-file`。
- 如果换了岗位、换了必须字段、或抽取字段不够，再全量重跑。

示例：

```bash
python3 scripts/resume_screening_pipeline.py score-only \
  --resumes ./resumes \
  --jd ./job_requirements.md \
  --work ./work \
  --output ./results \
  --workers 2
```

## 全量前确认语

在全量前向用户确认：

```text
我会按刚才确认后的 JD 和 pilot 反馈跑全量。
这次是【单一岗位 / 多岗位分流】。
推荐会偏【严格 / 平衡 / 宽口径】。
如果你确认，我开始处理全部简历。
```
