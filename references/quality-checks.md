# 质量检查

最终交付前做这些检查。

## 数据完整性

- 简历 inventory 数量和处理记录数量一致，除非有明确跳过的不支持文件。
- 每条记录都有 `candidate_id`、`source_file`、`extract_status`、`screen_status`。
- 没有重复 `candidate_id`。
- `candidate_index.json` 中同一文件 hash 在改名或新增文件后仍保持原 Candidate ID。
- `screening_summary.csv` 行数等于处理记录数。
- `screening_evidence.csv` 行数等于处理记录数。

## 失败处理

- 对 `extract_error`、`screen_error`、空抽取、无效 JSON 记录低并发重试一次。
- 重试后仍不可读或模糊的，标为 `需复核`。
- 错误信息保留在 JSON 记录里，方便复查。

## 排名合理性

- 每个标签至少抽查 3 个候选人。
- 检查 `推荐` 是否满足 must-have，或是否有清楚的例外理由。
- 检查 `不推荐` 是否只是因为缺少加分项而被误杀。
- 如果用户填写了人工初筛结果，确认重评后的 `AI 初筛结果` 和反馈方向一致，或模型给出了清楚的保留原判断理由。
- 检查 `AI 初筛结果` 和匹配证据是否一致。
- 检查证据是否来自简历正文，而不是文件夹名或来源渠道。
- 多岗位场景下，检查 best-fit role 是否合理。
- 检查邮件标题只用于投递岗位路由，没有被当作候选人能力证据。

## 交付物

- Excel 能打开，并包含 `筛选总表`、`使用说明` 和 `详细证据表`。
- `筛选总表` 的 `人工初筛结果` 和 `人工初筛判断依据` 两列位置明显、可编辑，并且用户知道保存 Excel 后可以让 agent 读取并重评。
- 分类文件夹数量和标签分布一致。
- 文件名已清洗，不超过系统限制。
- 公开示例不含真实候选人联系方式、API key、邮箱来源或私有公司信息。
- CSV 和 Excel 中以 `= + - @` 开头的外部文本已做公式注入转义。
- 用 pilot 的 Excel 运行 `score-only` 时，没有处理尚未存在缓存的简历。
- 不要公开上传 `work/records/`、`work/all_records.json` 或真实运行的结果表；隐私模式下这些本地文件仍可能保留联系方式映射。
