# 模型配置说明

## 默认推荐

默认推荐“便宜/限免模型抽取 + 更强模型判断”：

- 抽取层：GLM 或其他低成本模型，用来读简历、OCR、抽取结构化事实。
- 判断层：更强模型，用来对照 JD 做最终推荐判断。

这样做的原因是：几百份简历直接交给最强模型会更贵、更慢，也更容易撑爆上下文。先抽取再判断，更适合批量处理。

## 能不能直接用 coding agent 自己的模型

可以，但要分两种情况：

1. 小批量或人工协作场景：可以让 coding agent 自己读几份简历、做 pilot、解释结果。
2. 大批量自动化场景：不建议完全依赖对话里的 coding agent 模型逐份手工处理。

原因：

- 脚本无法天然调用“当前聊天里的 agent 模型”，除非平台提供可调用 API。
- 对话式处理几百份简历更慢，也更容易受上下文长度影响。
- 不如 API 调用容易并发、缓存、失败重试和审计。
- 如果 agent 中途断线，恢复粒度不如 `records/C###.json` 清楚。

使用建议：

如果你的 coding agent 已经自带可调用模型 API，可以把它配置成 `EXTRACT_MODEL` 或 `SCREEN_MODEL`；如果只是聊天窗口里的 agent，就适合做小样本试跑、复核和调口径，不适合承担几百份简历的全量自动化调用。

## 关于免费模型和成本

GLM / Z.AI / OpenRouter 等平台的免费或限免政策会变化。文档中不要承诺“永久免费”。更稳妥的说法是：

- 默认可使用低成本或限免模型做抽取。
- 成本通常不是主要瓶颈，时间、限流、失败重试和文件质量更影响体验。
- 如果用户使用公司内部模型或中国 coding agent 自带模型，需确认是否支持 API 调用、并发、结构化输出和简历数据合规。

## 时间预期

实际耗时取决于模型速度、简历页数、扫描件比例、网络、并发数和限流。

粗略预期：

- 5 份 pilot：通常几分钟内。
- 50 份：普通并发下适合短时间内完成。
- 200-300 份：建议预留一段较长时间，先跑 pilot，再用 `--workers 2` 或 `--workers 4`；中断后可续跑。

不要承诺固定分钟数。更适合承诺：可中断续跑、可失败重试、已完成的不重复付费/不重复处理。
